# SPEC.md - 结果库与节点交互体验优化

## 1. Objective

### 1.1 背景

当前项目已经具备 AI 生图、AI 生视频、视频拆帧、抠图、精灵图合成和 GIF 导出的完整工作流，但在长期使用体验上还存在三个明显问题：

1. 各类产出结果分散在不同节点内，缺少统一的记录、预览和复用入口。
2. 节点展开/收缩交互过于激进，点击画布空白会把节点收起，打断操作。
3. 一些 AI 节点在收缩后再次展开时，会丢失上传图片、参数或结果等 UI 状态。

### 1.2 目标用户

- 独立游戏开发者
- 动画师 / 像素画创作者
- 需要反复尝试 AI 素材并复用结果的单人创作者

### 1.3 本次要解决的核心目标

#### G1. 建立统一结果库

系统应当有“应该记录结果的地方”，统一记录以下四类产出：

- AI 生图结果
- AI 生视频结果
- GIF 导出结果
- 精灵图结果

结果记录需要同时满足以下四点：

- 页面内可见：在当前页面存在一个统一结果库 / 历史面板
- 浏览器持久化：刷新页面后仍能恢复
- 本地文件留档：支持保存到项目目录中的本地结果文件夹
- 可复用：结果可重新送回工作流中的后续节点继续加工

#### G2. 改善节点展开/收缩交互

节点展开与收缩必须变得“可控”，不能再因为点击画布空白而自动折叠，打断操作流。

#### G3. 保留节点 UI 状态

节点收缩前的状态，在再次展开后必须保持不变，尤其是：

- 已上传的参考图 / 首帧图 / 输入图
- prompt 与参数滑块
- 已生成结果
- 已选中的结果项
- 当前节点摘要和状态

#### G4. 减少 AI 生图与图生图能力重复

AI 生图节点应成为统一图片生成入口：

- 无输入图时：文生图
- 有输入图时：图生图

现有 `img2img` 节点可保留兼容旧流程，但不再作为新增节点的主入口。

### 1.4 验收标准

#### A. 结果库

1. 每次成功生成图片、视频、GIF、精灵图后，都会新增一条结果记录。
2. 结果记录至少包含以下字段：
   - `id`
   - `type`（image / video / gif / sprite）
   - `createdAt`
   - `sourceNodeId`
   - `sourceNodeType`
   - `prompt` 或关键参数摘要
   - `previewUrl`
   - `downloadUrl` 或 `filePath`
   - `reusePayload`
3. 页面中存在统一结果面板，可查看、预览、下载、重新送入工作流。
4. 刷新页面后，结果面板仍能恢复浏览器侧历史。
5. 当本地服务可用时，结果文件会被写入项目内固定目录，并维护索引清单。

#### B. 节点交互

1. 点击节点标题，只展开/收缩当前节点。
2. 点击画布空白，不自动收缩任何已展开节点。
3. 如需批量收起，提供明确入口（按钮或菜单项）。
4. 多个节点可以同时保持展开，不强制只保留一个。

#### C. 状态保留

1. 节点收起后再次展开，上传图片仍存在。
2. 节点收起后再次展开，prompt、参数、缩略图、结果列表仍存在。
3. 节点重新绑定 UI 时，不得把已有 runtime 状态重置为空。

#### D. 能力整合

1. `aiImage` 节点支持接收上游 `image` 输入。
2. `aiImage` 节点在有参考图或上游图片时，走图生图模式。
3. 右键新增节点菜单中不再主推单独 `img2img` 节点。
4. 已保存工作流中若仍存在 `img2img` 节点，仍可继续运行。

## 2. Commands

### 2.1 开发 / 运行命令

```bash
# 启动本地工具服务
node server.js

# 浏览器访问
http://localhost:3000
```

### 2.2 结果库相关用户动作

结果库至少支持以下用户动作：

- 查看结果列表
- 预览结果
- 下载原始文件
- 复制 / 重新送入后续节点
- 定位结果来源节点
- 清理单条记录（仅前端记录时）
- 打开本地保存目录（服务端可用时）

### 2.3 建议新增服务端接口

以下接口为本次规范的推荐方案，用于支撑本地结果留档。当前阶段先写入规范，不要求立刻全部实现。

#### 结果写入

- `POST /api/results/image`
- `POST /api/results/video`
- `POST /api/results/gif`
- `POST /api/results/sprite`

请求体建议包含：

```json
{
  "sourceNodeId": "aiImage_1",
  "sourceNodeType": "aiImage",
  "prompt": "像素风角色",
  "summary": "4 张图片",
  "createdAt": "2026-05-06T15:10:00.000Z",
  "data": {
    "urls": ["..."],
    "dataUrl": "...",
    "meta": {}
  }
}
```

#### 结果索引读取

- `GET /api/results`

返回结果清单，用于页面中的“结果库”面板。

#### 本地文件打开（可选）

- `POST /api/results/open-folder`

仅在用户主动点击时执行，不自动打开。

## 3. Project Structure

### 3.1 前端主要改动点

文件：`sprite-tool.html`

需要重点修改以下区域：

1. **全局状态 `S`**
   - 新增统一结果库状态，如 `S.resultLibrary`
   - 区分“结果预览 URL”和“下游处理安全 URL”

2. **节点 runtime / data 存储**
   - 扩展 `ensureNodeRuntime(node)`
   - 明确区分：
     - `config`：用户参数
     - `runtime`：上传文件、已生成结果、UI 选择状态
     - `data.inputs / data.outputs`：节点间传输数据

3. **节点展开逻辑**
   - 移除“点击画布空白自动收缩全部节点”行为
   - 保留显式折叠按钮和新增“全部收起”入口

4. **AI 节点绑定逻辑**
   - `bindAiImage()` 改为支持文生图 / 图生图统一入口
   - `bindImg2Img()` 保持兼容旧节点
   - 所有上传图片状态必须从 runtime 恢复，而不是每次 bind 时重置

5. **结果库 UI**
   - 在主界面新增结果库面板或抽屉
   - 能显示图片、视频、GIF、精灵图四类结果
   - 每条结果具备“预览 / 下载 / 复用”操作

6. **持久化 `PM`**
   - 现有布局保存继续保留
   - 额外为结果库增加本地持久化键，例如：`spritetool_result_library`
   - 节点 UI 的必要 runtime 状态，按需写入持久化结构

### 3.2 服务端主要改动点

文件：`server.js`

需要重点修改以下区域：

1. **结果文件保存目录**
   - 建议目录：

```text
results/
  manifest.json
  images/
  videos/
  gifs/
  sprites/
```

2. **结果保存接口**
   - 接收前端传来的结果数据
   - 落盘文件
   - 更新 `manifest.json`

3. **静态访问**
   - 本地结果文件应可被页面安全读取与预览

4. **图片/视频代理**
   - 图片代理用于下游处理稳定读取
   - 预览优先直链，处理优先代理

### 3.3 推荐数据结构

#### 前端结果记录结构

```javascript
{
  id: 'result_20260506_001',
  type: 'image',
  createdAt: '2026-05-06T15:10:00.000Z',
  sourceNodeId: 'aiImage_1',
  sourceNodeType: 'aiImage',
  prompt: '像素风角色',
  summary: '4 张图片',
  previewUrl: 'https://...',
  downloadUrl: 'https://... 或 /results/...',
  filePath: 'results/images/...png',
  reusePayload: { urls: ['...'] },
  meta: {}
}
```

#### 节点 runtime 建议

```javascript
node.runtime = {
  uploadedImages: [],
  generatedResults: [],
  selectedResultId: null,
  formState: {},
  aiThumbs: [],
  processedFrames: []
}
```

## 4. Code Style

1. 保持单文件前端架构，不引入框架。
2. 优先在现有 `state / NM / CM / PM` 结构上增量修改，不推倒重来。
3. 节点状态恢复优先使用明确字段，不依赖 DOM 当前值反推。
4. 结果记录结构必须统一，图片 / 视频 / GIF / 精灵图都使用同一套基础字段。
5. 不新增无必要抽象；同一逻辑在 2-3 个节点复用时再提炼 helper。
6. 默认不写注释；只有在状态恢复或文件持久化规则存在隐含约束时，才写简短 why 注释。
7. 不使用 `alert()`，继续使用 toast。
8. 不为兼容旧逻辑保留长期“双轨状态”；能收敛的状态尽量收敛到 `runtime` 与结果库。

## 5. Testing Strategy

### 5.1 手工验收主路径

#### T1. AI 生图结果可见与可复用

1. 打开 `AI 生图` 节点
2. 生成图片
3. 结果区立即显示新图片
4. 结果库新增对应记录
5. 点击结果库中的“复用”，能送入支持 `image` 输入的节点

#### T2. 图生图结果可继续传给抠图

1. 使用 `AI 生图` 的上游输入模式或旧 `img2img` 节点生成图片
2. 连到 `抠图` 节点
3. 抠图节点预览区能正常显示输入图
4. 批量处理成功

#### T3. 图生图 / 生图结果可直接进入导出节点

1. 将图片结果连接到 `导出`
2. 可下载 PNG
3. JSON 按钮对非精灵图输入给出明确 warning，不静默失败
4. GIF 导出对单张图片至少能生成单帧 GIF，或对多张图片生成多帧 GIF

#### T4. AI 生视频结果记录

1. 成功生成视频
2. 节点内可预览
3. 结果库新增视频记录
4. 可从结果库重新送入 `导入视频` 节点

#### T5. 精灵图 / GIF 结果记录

1. 成功生成精灵图
2. 成功导出 GIF
3. 两者都进入结果库
4. 刷新后仍能恢复记录

#### T6. 节点展开交互

1. 展开多个节点
2. 点击画布空白
3. 所有节点保持展开，不自动折叠
4. 点击节点自身折叠按钮，仅折叠当前节点

#### T7. 节点状态保留

对以下节点至少逐一验证：

- `aiImage`
- `aiVideo`
- `removeBg`
- `importVideo`

验证步骤：
1. 上传图片或设置参数
2. 生成结果或进入处理中状态
3. 收起节点
4. 再展开
5. 上传图、参数、结果、选中状态仍然存在

#### T8. 刷新恢复

1. 页面生成若干结果
2. 刷新浏览器
3. 结果库恢复
4. 已保存布局继续存在
5. 不要求所有处理中临时状态都恢复，但已完成结果必须恢复

### 5.2 服务端验收

1. 本地结果目录自动创建
2. 四类结果分别保存到正确子目录
3. `manifest.json` 正确追加记录
4. 图片 / 视频代理不影响原有 AI 工作流

### 5.3 回归重点

- AI 生视频下载与导入不能回退
- 选择帧的选中结果不能回退成全量传递
- `sprite -> image -> removeBg` 链路不能回退
- 页面刷新后节点脚本不能再出现空白画布问题

## 6. Boundaries

### 6.1 Always do

1. 所有图片、视频、GIF、精灵图产出都必须通过统一结果记录入口入库。
2. 节点重新展开时优先恢复既有 runtime 状态，不能先清空再重建。
3. 结果预览与下游处理要分离：
   - 预览优先使用最直接可显示的 URL
   - 下游像素处理优先使用稳定可读的代理 URL
4. 旧流程能跑通时，优先做最小改动，不整体重写工作流系统。

### 6.2 Ask first about

1. 是否要物理删除本地结果文件
2. 是否要清空历史结果库
3. 是否要迁移旧的 `img2img` 节点实例
4. 是否要改变结果保存目录结构

### 6.3 Never do

1. 不自动清空用户已有结果记录
2. 不在点击画布空白时隐式收起全部节点
3. 不在节点重新绑定时无条件重置上传图片状态
4. 不为了省事移除现有旧节点兼容能力
5. 不引入数据库或额外前端框架

---

## 实施建议顺序

1. 先修正节点展开/收缩行为
2. 再统一节点 runtime 状态恢复
3. 再补前端结果库 + localStorage 持久化
4. 最后补 server.js 的本地结果落盘与清单接口

---

## 当前结论

本次改造的本质不是新增单个功能，而是把“结果资产管理”和“节点交互状态管理”提升为一等能力。

优先级如下：

1. 节点不要乱收起
2. 节点收起后状态不能丢
3. 所有产出结果必须统一可见、可复用、可持久化
4. `aiImage` 成为统一图片生成入口，`img2img` 转为兼容旧流程的保留节点
